User Set up / Registration  
Q: What is the difference between Login and Registration?  
A: An Agent or Policy Owner must Register to create an account on the site. This requires them to input information to identify the agent ID or policy number they will have authority to view.  
An Agent or Policy Owner only needs to register once (unless they have not completed their registration). Once they have entered their user information and confirmed their phone number by successfully entering the Verification Code at Registration, they are considered as having completed their

--- PAGE 2 ---

registration and would use the credentials they created at Registration to Login.

For Home Office accounts, who are not accessing only specific Agent or Policy data, but are authorized to see all Agents and Policies in the Client's block of business, users do not Register. Their accounts are administratively established. Home Office users will always ONLY Login - never Register.

Note: There are a few LBL employees as well who are set up as Home Office users. Like DXC Home Office users, these accounts are administratively created and will never use the Registration function, only Login.

Refer to Q: How do I request an account, as a DXC Employee? for information on how to request an account for a DXC Employee.

Q: What is an account?

A: An account is the unique user information which is created for a person who accesses the web site(s). It is always a person who is creating an account either representing themself as an individual or on behalf of an entity they represent, e.g., an Agency for the Agent site or a Trust or Business for a Policy Owner.

DXC Employees do not register - their accounts are administratively set up.

Accounts are composed of first name, last name, username, email, password, and phone number (must be capable of receiving text messages). A Policy Owner account is associated with one or more policies. An Agent account is associated with one or more Agent IDs.

Q: How do I request an account, as a DXC Employee?

A: DXC Call Center, Policy Holder Services, Agency Services and IT Support team members may be granted Home Office user access to both sites. For those who previously had access to the site, your managers were asked to rectify your access. Users who were confirmed as needing access have been set up on the new site. If you believe you should be granted access, your manager must submit the access request for you into LifeBPO Service Now (https://lifepolhedpesk.service-now.com/) using LAN Production access request form (make sure to use the April 2026 version of the form or later, else the request will be rejected) and completing the info per the User Guide tab on the User Info and Portal tabs. 

Note: LBL also has a small group of employees who are set up as Home Office users as well. Their requests will come into the team providing Managed Services Support to Agent and Owner Portals directly.

Q: How does a Policy OWNER create an account (register)?

A: Policy Owners must create their own account via the Registration process.

Steps for registration are as follows:

1. From the initial "landing" page, users will select Registration / Login for Policy Owners.
2. To create their account, users must click the Register Account button which will guide them through the dialog to create an account.

1. The first two pieces of information that the Policy Owner must provide are the policy number and the last four digits of the policy owner's SSN/TIN. The policy must be found in the uniA system (which retrieves its data from CyberLife feeds as part of every cycle run) and the SSN/TIN of the policy owner uniA must match the last four digits entered by the user. The user does need to enter the last four digits of the SSN/TIN twice to confirm their entry. User must take care to enter the policy number exactly as it is found on their Insurance Contract. Common issues seen which cause problems can be when a user enters a zero instead of the letter "O" or vice versa. Similar issues can occur with number one and letter "l." Only ONE user may be registered as the policy owner, even for joint owned policy. If the policy has already been registered by one co-owner, the additional co-owner would not be able to proceed from verification of the policy number and SSN. If the policy number has already been registered by another user, they will receive this error message: Our records indicate you already have an existing account.

Transitioning from the previous LBL site
Q: What's changed?

A: The new site ...

1. is AWS hosted, aka "on the cloud",
2. uses modern frameworks,
3. is responsive (meaning the UI will resize depending on the form factor of the device running the browser),
4. is built upon the DXC Assure Platform, and
5. implements multifactor authentication (MFA) for user access (e.g., Verification Code must be input as well as user credentials (username and password)). MFA requires us to capture a phone number for each user. This is a change from the prior site. This is required to be able to send a text over SMS with their Verification Code for certain transactions. Specifically, registration, change Password and Change Email require the user to receive the Verification Code via text.

The previous site hosted on the Norwich data center servers was used as the foundation for functionality. Other than the addition of MFA and WEBREG creation and removal of Impersonation and Agent Staff account creation, business functionality is the same as the previous site. The technologies used to implement the site, however, are totally different so it was a complete rewrite.

Q: Can I use my credentials (e.g., username and password) from the old site on the new site to login? 

A: No. Agent and Policy Owner users from the "old" site need to re-register on the new site. The new site provides greater security by requiring multi-factor authentication (MFA). A: Agent and Policy Owner users need to re-register on the new site as the new site provides greater security. In order to supply the Verification Code to users, we must ensure we have a current phone number for the user which is capable of receiving SMS (text) messages to provide the Verification Code via text (the previous site did not collect user's phone numbers), and we must ensure we have the most current email address for the user.

Q: How do I impersonate a user?

A: The AWS hosted site does NOT have the capability for a Home Office user to Impersonate an Agent or Policy Owner.

Q: How does a user link additional Policies to their user account?

A: Some registered users may be an Owner on multiple policies. They should not try to create a separate account for each policy. Instead, a registered policy owner user can link additional policies to their existing user account by navigating to Account Management and selecting Linking Policy. The policy number must be known and the owner SSN/TIN must match that of the originally registered policy for the user. After entering the required information, the user must Submit the request. The system will attempt to locate the policy in uniA and display the policy details. If the policy number entered is found and confirmed as owned by the user, the policy will be linked and displayed under Policies Linked To Your Account. If the user specifies a policy number that is not found or does NOT match the SSN/TIN of their original registration, they will see the error "Invalid Credentials. Please contact our Service Center at 1-844-768-6777, Monday-Friday 8:00 AM - 5:00 PM CT, for assistance."

Q: Can a Policy Owner register or link a policy that is suspended or restricted?

A: Yes. Registration and linking of a policy are allowed when the policy is suspended or restricted, however, the Policyowner will only see a subset of their policy data on the screen.

Q: Can I submit transactions on behalf of the policy owner?

A: No. As a Home Office User, you will be able to see the screens the Policy Owner sees for submitting address changes or billing updates, but you will not have the Submit option enabled. Only the registered user as Policy Owner can proceed with the Submit step.

Q: Is data retrieved from Cyberlife realtime?

A: No. The policy data comes from uniA. Each run of Cyberlife cycles will send a feed to uniA if there were policy updates for that day. uniA will load the feed to update the uniA databases. The data is read realtime from uniA databases.

Q: Are update transactions processed into Cyberlife realtime?

A: No. The user registered as the Policy Owner may submit a request for an address change and/or a billing change, but these only create ADDCHG and BILCHG work items in AWD which are then addressed by PHS. The request has to be deemed as IN GOOD ORDER (IGO) by PHS and then the respective update applied in Cyberlife.

Q: Can a Policy Owner change their username?

A: No. The username cannot be changed once the account is created.

Q: Can a husband and wife, each with their own separate policy, register using the same phone number?

A: No. Different site users cannot be registered using the same SMS-enabled phone number. The phone number is treated as part of the user's unique identifying information and therefore cannot be shared between multiple user accounts.

Troubleshooting

What do I do if I need Support?  
Q: Who do I contact for support?

A: Remember to always gather as much info as you can when reporting an issue, including screen prints if able. Remember to raise the concern with your lead to make sure a ticket is appropriate. Leads should ensure that the behavior is abnormal before raising a ticket.  
An OUTAGE indicates the site is down or the main functions are not working overall (e.g., No users can login).

Outages should be raised in BPO Service Now, https://lifebohelpldesk.service-now.com/. Select client as Resolution Life (LBL-RSL) and the appropriate impacted application, Agent Portal or Policy Owner Websites.

A DEFECT indicates there is an issue for a specific user or policy or agent. Defects are typically more limited in scope than an Outage, e.g., specific policies or users are experiencing an issue.

A DATA CORRECTION for certain data fields for a user account can also be submitted via Jira. Note: The only things we can correct are phone number, email, and first and last name. Username cannot be modified once the account has been created.

Defects and DATA CORRECTIONS should be reported in Jira, https://jira.dxc.com/, within the LBL Production project. Call Center leads have been granted access to the Jira project and can submit tickets for their team when appropriate. As a Call Center lead, when entering a Jira ticket, the Component selected should be either Agent Portal or Owner Portal and select Issue type as Help Desk Request.

Miscellaneous Info  
Q: Can a user access either site if they registered with a landline phone?  
A: No. As part of their registration, the user must specify a phone number which is capable of receiving SMS text messages in order to receive their Verification Code. The registration process itself does not have any way to determine whether the phone number the user provides is capable of receiving SMS texts.  
▪ Note: If the user is unable to complete registration due to entering an incorrect phone number or a number that cannot receive SMS text messages, the user profile will be automatically deleted overnight (1:00 AM CST). If this occurs, the user will need to re-register on the next business day, and specify a phone number that supports SMS text messaging.  
Q: What is the user session timeout?

A: Users who have been inactive for 15 minutes will have their session logged out. They will need to log back in after that time to regain access the site.  
Q: Do the terms Verification Code, MFA Code, and One-Time Passcode (OTP) all mean the same thing?  

A: Yes. For the purpose of the LBL sites, the terms Verification Code, MFA Code, and One-Time Passcode (OTP) may be considered interchangeable. IT selected to use the term Verification Code as what is presented to the user on the site, in texts and emails.
